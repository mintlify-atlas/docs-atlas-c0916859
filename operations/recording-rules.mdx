## Why Recording Rules?

Recording rules serve two primary purposes:

- **Performance**: Expensive queries (for example, CPU utilization across many CPUs) are precomputed and stored as new time series, so dashboards and alerts load instantly instead of recomputing across large datasets on every query.
- **Normalization**: Recording rules let you define canonical aggregated metric names, making it easier to build dashboards and alerts that remain consistent across environments.

The node_exporter ships with example recording rules in [`example-rules.yml`](https://github.com/prometheus/node_exporter/blob/master/example-rules.yml) that follow the Prometheus [recording rules naming convention](https://prometheus.io/docs/practices/rules/): `level:metric:operations`.

## Adding Recording Rules to Prometheus

<Steps>
  <Step title="Create a rules file">
    Create a file (e.g. `/etc/prometheus/rules/node_exporter.yml`) with the recording rules shown below.
  </Step>
  <Step title="Reference the file in prometheus.yml">
    Add a `rule_files:` entry pointing to your rules file (or a glob that matches it):

    ```yaml prometheus.yml
    global:
      scrape_interval: 15s

    rule_files:
      - /etc/prometheus/rules/*.yml

    scrape_configs:
      - job_name: 'node'
        static_configs:
          - targets: ['localhost:9100']
    ```
  </Step>
  <Step title="Reload or restart Prometheus">
    Send a `SIGHUP` to Prometheus or call the reload endpoint to pick up the new rules without a full restart:

    ```bash
    # Signal-based reload
    kill -HUP $(pidof prometheus)

    # HTTP reload (requires --web.enable-lifecycle flag)
    curl -X POST http://localhost:9090/-/reload
    ```
  </Step>
  <Step title="Verify the rules loaded">
    Check the Prometheus UI at `http://localhost:9090/rules` or query the new recording rule metrics in the expression browser to confirm they are being evaluated.
  </Step>
</Steps>

## CPU Recording Rules

The following recording rules are taken directly from the node_exporter [`example-rules.yml`](https://github.com/prometheus/node_exporter/blob/master/example-rules.yml):

<CodeGroup>

```yaml node_exporter_recording_rules.yml
groups:
  - name: example-node-exporter-rules
    rules:
      # The count of CPUs per node, useful for getting CPU time as a percent of total.
      - record: instance:node_cpus:count
        expr: count(node_cpu_seconds_total{mode="idle"}) without (cpu,mode)

      # CPU in use by CPU.
      - record: instance_cpu:node_cpu_seconds_not_idle:rate5m
        expr: sum(rate(node_cpu_seconds_total{mode!="idle"}[5m])) without (mode)

      # CPU in use by mode.
      - record: instance_mode:node_cpu_seconds:rate5m
        expr: sum(rate(node_cpu_seconds_total[5m])) without (cpu)

      # CPU in use ratio.
      - record: instance:node_cpu_utilization:ratio
        expr: >
          sum(instance_mode:node_cpu_seconds:rate5m{mode!="idle"}) without (mode)
          / instance:node_cpus:count
```

```yaml node_exporter_alerts.yml
groups:
  - name: node-exporter-alerts
    rules:
      # Node is unreachable
      - alert: NodeDown
        expr: up{job="node"} == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Node {{ $labels.instance }} is down"
          description: >
            Node exporter on {{ $labels.instance }} has been unreachable
            for more than 2 minutes.

      # High CPU utilization using precomputed recording rule
      - alert: HighCPUUsage
        expr: instance:node_cpu_utilization:ratio > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High CPU usage on {{ $labels.instance }}"
          description: >
            CPU utilization on {{ $labels.instance }} has been above 90%
            for more than 5 minutes (current: {{ $value | humanizePercentage }}).

      # High memory usage
      - alert: HighMemoryUsage
        expr: >
          1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage on {{ $labels.instance }}"
          description: >
            Memory usage on {{ $labels.instance }} has been above 90%
            for more than 5 minutes (current: {{ $value | humanizePercentage }}).

      # Disk nearly full
      - alert: DiskNearlyFull
        expr: >
          (
            1 - (
              node_filesystem_avail_bytes{fstype!~"tmpfs|overlay|aufs|squashfs"}
              / node_filesystem_size_bytes{fstype!~"tmpfs|overlay|aufs|squashfs"}
            )
          ) > 0.85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Disk nearly full on {{ $labels.instance }}"
          description: >
            Filesystem {{ $labels.mountpoint }} on {{ $labels.instance }} is
            {{ $value | humanizePercentage }} full.
```

</CodeGroup>

## Recording Rule Reference

| Recording Rule | Expression | Description |
|---|---|---|
| `instance:node_cpus:count` | `count(node_cpu_seconds_total{mode="idle"}) without (cpu,mode)` | Number of CPUs per instance |
| `instance_cpu:node_cpu_seconds_not_idle:rate5m` | `sum(rate(node_cpu_seconds_total{mode!="idle"}[5m])) without (mode)` | CPU seconds not idle, per CPU, over 5m |
| `instance_mode:node_cpu_seconds:rate5m` | `sum(rate(node_cpu_seconds_total[5m])) without (cpu)` | CPU seconds per mode over 5m |
| `instance:node_cpu_utilization:ratio` | `sum(instance_mode:node_cpu_seconds:rate5m{mode!="idle"}) without (mode) / instance:node_cpus:count` | CPU utilization as a 0–1 ratio |

## Alert Rule Summary

| Alert | Condition | Severity |
|---|---|---|
| `NodeDown` | `up{job="node"} == 0` for 2m | critical |
| `HighCPUUsage` | CPU utilization > 90% for 5m | warning |
| `HighMemoryUsage` | Memory used > 90% for 5m | warning |
| `DiskNearlyFull` | Disk used > 85% for 5m | warning |
